# Secret Handling — FrostGate Core

Operational reference. No fluff.

---

## How secrets are stored

Secrets are **not stored in this repository**.

`env/prod.env` is tracked as a **template only**.  Every secret-class variable
in that file contains a `CHANGE_ME_<VAR_NAME>` placeholder.  A placeholder is
not a default — it is an explicit marker that the variable has not been
injected yet and **must be rejected at runtime** in production environments.

---

## How secrets are injected

Injection happens at deploy time, **outside** the repository:

1. **Local development** — copy `env/prod.env` to a gitignored file, populate
   real values, and export them before starting services.  Never commit the
   populated file.

2. **CI** — ephemeral secrets are generated by
   `.github/actions/fg-secrets/action.yml` at job start.  These values are
   discarded when the runner terminates.

3. **Production / staging** — secrets are fetched from a secrets manager
   (Vault, AWS SSM, GCP Secret Manager, or equivalent) and injected as
   environment variables before the container starts.  The `env/prod.env`
   template is never populated on production hosts.

---

## What `CHANGE_ME_*` means

A `CHANGE_ME_*` value is a sentinel, not a default.

- The runtime (`api/config/required_env.py`) treats `CHANGE_ME_*` as missing.
- Any prod/staging startup with a `CHANGE_ME_*` value for a required variable
  **will fail hard** with `RuntimeError`.
- CI (`tools/ci/check_no_plaintext_secrets.py`) will **fail the PR** if any
  secret-class variable contains a non-placeholder value.

## What causes CI failure

| Gate | File | Trigger |
|---|---|---|
| Secret scanning gate | `tools/ci/check_no_plaintext_secrets.py` | Any secret-class variable in `env/*.env`, `.env.example`, `agent/.env.example` contains a non-`CHANGE_ME_*` value |
| Secret history audit | `tools/ci/check_secret_history.py` | A known-leaked literal reappears in any non-exempt file in current HEAD |
| Runtime startup | `api/config/required_env.py` | `CHANGE_ME_*` or blank value for a required var in prod/staging |
| Pre-commit hook | `.pre-commit-config.yaml` | Same as secret scanning gate, runs locally before commit |

## How to fix CI failures

### `check_no_plaintext_secrets` fails

```
FAIL: plaintext secrets detected in env files:
  env/prod.env:11: 'POSTGRES_PASSWORD' has a non-placeholder value
```

1. Replace the value: `POSTGRES_PASSWORD=CHANGE_ME_POSTGRES_PASSWORD`
2. Rotate the real credential externally.
3. Inject the rotated value at deploy time — never commit it.

### `check_secret_history` fails (HEAD violation)

A blocked literal was re-introduced into a non-exempt file.

1. Find the file and remove the literal.
2. Replace with the appropriate `CHANGE_ME_*` placeholder.
3. If the literal was accidentally staged, use `git reset HEAD <file>`.

### Runtime `RuntimeError: Missing required production env vars`

A required variable is absent, blank, or still set to `CHANGE_ME_*`.

1. Inject the real secret via your deployment pipeline.
2. Verify with `echo $VAR_NAME` in the container before starting the app.

## Adding a new secret

1. Add the variable to `env/prod.env` with a `CHANGE_ME_<VAR_NAME>` value.
2. If required at runtime, add it to `REQUIRED_PROD_ENV_VARS` in
   `api/config/required_env.py` and update `_VALID_PROD_ENV` in
   `tests/security/test_required_env_enforcement.py`.
3. The secret-class name patterns in `tools/ci/check_no_plaintext_secrets.py`
   cover all standard suffixes (`_PASSWORD`, `_SECRET`, `_TOKEN`, `_KEY`,
   `_CREDENTIAL`, etc.).  If the new variable uses a non-standard name, add
   the suffix to `_SECRET_SUFFIXES` in that file.
4. Document the variable in the rotation checklist above.
